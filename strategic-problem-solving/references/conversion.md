# Rutas de conversión

Después de las recomendaciones, indicar al usuario qué artefacto downstream conviene y ofrecer generarlo.

---
## Convertir a PRD cuando
- El usuario/cliente objetivo está claro.
- Las métricas de resultado están claras.
- La dirección de solución es lo bastante estable para especificar.
- Los requisitos se pueden expresar como capacidades, workflows, necesidades de datos y criterios de aceptación.

```markdown
# PRD Outline
- Background
- Problema
- Usuarios/stakeholders objetivo
- Objetivos y no-objetivos
- Solución propuesta
- Requisitos funcionales
- Requisitos no funcionales
- Requisitos de datos
- Riesgos y supuestos
- Métricas de éxito
- Plan de rollout
- Preguntas abiertas
```

---
## Convertir a presentación ejecutiva cuando
- La necesidad principal es alineación de stakeholders, sponsorship, funding o apoyo político.

```markdown
# Presentación Ejecutiva Outline
1. Contexto y problema
2. Por qué ahora
3. Diagnóstico
4. Escenarios
5. Opciones consideradas
6. Camino recomendado
7. Impacto esperado
8. Riesgos y mitigaciones
9. Roadmap
10. Decisión requerida
```

---
## Convertir a propuesta comercial cuando
- El objetivo es vender o formalizar un engagement de consultoría/servicio.

```markdown
# Propuesta Comercial Outline
- Contexto del cliente
- Problema/oportunidad
- Enfoque propuesto
- Workstreams
- Entregables
- Timeline
- Inputs requeridos del cliente
- Métricas de éxito
- Governance
- Inversión / placeholder de pricing
- Supuestos y exclusiones
- Próximos pasos
```

---
## Convertir a backlog/specs cuando
- Hay claridad suficiente para ejecutar y descomponer el trabajo.

```markdown
# Backlog de Ejecución Outline
- Épica
- Objetivo
- Usuario/historia/job
- Requisitos funcionales
- Criterios de aceptación
- Requisitos técnicos/de datos
- Dependencias
- Tests / validación
- Definition of done
```
